How the Web Works

The Client-Server Dance

💻

Client (Browser)

Requests resources

Chrome, Firefox, Safari

→ HTTP Request "GET index.html" ← HTTP Response HTML, CSS, JS files
🖥️

Server

Sends resources

Apache, Nginx, Node.js

The Journey of a Web Request:

  1. You type: www.google.com
  2. DNS Lookup: "What's the IP address?" → 142.250.80.46
  3. HTTP Request: "Please send me your homepage"
  4. Server Processes: "Here's the HTML, CSS, and JavaScript"
  5. Browser Renders: Builds the page you see

Why HTML Came To Be

"In 1989, Tim Berners-Lee had a problem..."

The Problem:

Scientists at CERN had documents scattered across different computers with:

  • 🖥️ Incompatible systems: VAX, Unix, PCs, Macs - couldn't read each other's files
  • 📁 Different formats: WordPerfect, TeX, PostScript - no universal standard
  • 🔒 No linking: References to other papers required manual searching
  • 🌍 No remote access: Had to physically go to a computer to read its files
  • 📧 Email limitations: Could send text, but lost all formatting and structure

Tim Berners-Lee's insight: "What if documents could live on any computer, be read by any computer, and link to each other automatically?"

The Solution: HTML (1991)

HyperText Markup Language - A universal way to structure documents

1989
Problem identified
1991
HTML born
1993
First browser (Mosaic)
Today
HTML5

HTML's Original Goals:

  • 📄 Share documents between any computer
  • 🔗 Link documents together (hyperlinks!)
  • 📖 Be human-readable and simple
  • 🌍 Work on any operating system
<!-- Tim Berners-Lee's first HTML looked like this -->
<TITLE>The World Wide Web project</TITLE>
<H1>World Wide Web</H1>
<P>The project... <A HREF="hypertext.html">What's hypertext?</A>

Why CSS Came To Be

"By 1994, the web was ugly and every browser displayed pages differently..."

The Problem with Early HTML:

<!-- The nightmare of 1994 HTML styling -->
<FONT COLOR="red" SIZE="5" FACE="Arial">
  <B><I><U>Important Text</U></I></B>
</FONT>
<FONT COLOR="blue" SIZE="3" FACE="Times">
  Different styled text
</FONT>

Imagine doing this for EVERY piece of text on EVERY page! 😱

The Solution: CSS (1996)

Cascading Style Sheets - Separate content from presentation!

/* Write once, use everywhere! */
h1 {
  color: red;
  font-size: 32px;
  font-family: Arial;
}

CSS Solved:

  • 🎨 Consistency: One style sheet for entire website
  • 🔧 Maintenance: Change look of 1000 pages in one place
  • ♿ Accessibility: Screen readers ignore styling
  • 📱 Flexibility: Different styles for different devices

Speaking the Language: HTML & CSS Terminology

In HTML:

<h1>Hello World</h1>
  • <h1> is an opening tag
  • </h1> is a closing tag
  • The whole thing is an element
  • "Hello World" is the content

In CSS:

h1 {
  color: red;
  font-size: 32px;
  font-family: Arial;
}

CSS Anatomy:

  • 📍 h1 = Selector
  • 🎯 color = Property
  • 🎨 red = Value
  • 📝 color: red; = Declaration
  • 📦 { ... } = Declaration Block
  • 📋 Entire thing = CSS Rule

How They Connect:

The CSS selector targets the HTML element!

When CSS sees h1, it finds all <h1> elements in your HTML and applies those styles.

Say it like this:
"This CSS rule uses the h1 selector to style all h1 elements."

Remember: HTML has tags/elements, CSS has selectors/rules!

Why JavaScript Came To Be

"In 1995, websites were completely static. Click a button? Wait for a page reload..."

The Problem:

Every interaction required a full page reload from the server!

Form validation? → Send to server → Wait → Reload page → Show errors 😴

The Solution: JavaScript (1995)

Created in just 10 days by Brendan Eich at Netscape!

JavaScript Enabled:

  • ⚡ Instant feedback: Validate forms without server
  • 🎮 Interactivity: Respond to clicks, hovers, typing
  • 🔄 Dynamic content: Change page without reload
  • 🎯 User experience: Make web feel like desktop apps

Fun Fact:

Called "JavaScript" to ride the popularity of Java (they're not related!)

Original name was "LiveScript" → Marketing changed it to "JavaScript"

// The power of JavaScript - instant interaction!
button.addEventListener('click', function() {
  alert('No page reload needed!');
});

How HTML, CSS, and JavaScript Bundle Together

The Server Sends Three Types of Files:

Server Bundle

📄 index.html

🎨 styles.css

⚡ script.js

→ HTTP Response

Browser Receives

1. Parse HTML

2. Apply CSS

3. Execute JS

The Browser's Rendering Process:

  1. HTML arrives first: Browser builds the DOM (Document Object Model)
  2. CSS loads: Browser calculates styles for each element
  3. JavaScript executes: Can modify both HTML and CSS dynamically
  4. Page renders: You see the final result!
<!-- How they connect in HTML -->
<!DOCTYPE html>
<html>
<head>
  <link rel="stylesheet" href="styles.css">
</head>
<body>
  <h1>Content from HTML</h1>
  <script src="script.js"></script>
</body>
</html>

What the Server Actually Sends

It's Just Text Files!

The server sends plain text files that your browser interprets:

1. Initial Request

GET /index.html HTTP/1.1
Host: example.com
→

2. Server Response

HTTP/1.1 200 OK
Content-Type: text/html

<html>...</html>

The Browser Then:

  1. Reads the HTML and finds <link> and <script> tags
  2. Makes additional requests for CSS and JavaScript files
  3. Downloads images, fonts, and other resources
  4. Assembles everything into the page you see

Open DevTools to see this happen!

Windows/Linux: Ctrl+Shift+I or Right-click → Inspect

Mac: Cmd+Option+I or Right-click → Inspect

→ Click the "Network" tab → Refresh the page (F5)

You'll see 10-100+ requests for a single page load!

How the Browser Renders Everything

The Rendering Pipeline:

1. Parse HTML → Build DOM Tree

Creates a tree structure of all elements

2. Parse CSS → Build CSSOM Tree

Creates a tree of all styles

3. Combine → Create Render Tree

Matches styles to elements

4. Layout → Calculate Positions

Determines where everything goes

5. Paint → Draw Pixels

Actually displays on screen

This Happens FAST!

  • Modern browsers render 60 times per second (60 FPS)
  • JavaScript can trigger re-renders by changing DOM
  • CSS animations use GPU for smooth performance

This entire slide deck re-renders every time you change slides!

Git Fundamentals

The Basic Git Workflow:

# 1. Make changes to your files
# 2. Stage changes (choose what to save)
git add filename.html
git add . # adds everything

# 3. Commit (save snapshot with message)
git commit -m "Added navigation menu"

# 4. Push to GitHub (backup online)
git push

Think of it like:

  • Working Directory: Your desk where you're actively working
  • Staging Area: A box where you put finished work
  • Repository: Filing cabinet where you store everything
  • GitHub: Cloud backup of your filing cabinet

Commit Messages Matter!

❌ Bad:
"fixed stuff"
"asdfasdf"
"work"
✅ Good:
"Fixed navigation dropdown bug"
"Added contact form validation"
"Updated color scheme to match brand"

Creating and Managing Repositories

Repository Best Practices:

📁 Organization:

my-website/
├── index.html
├── css/
│ └── styles.css
├── js/
│ └── script.js
├── images/
│ └── logo.png
└── README.md # Always include this!

📝 README.md Essentials:

  • Project title and description
  • Technologies used
  • How to run/view the project
  • What you learned
  • Screenshots (add later)

🎯 Commit Frequently:

Don't wait until "perfect" - commit working progress!

  • After adding a new feature ✅
  • Before trying something risky ✅
  • At end of each work session ✅
  • Only when 100% done ❌

Golden Rule: Commit early, commit often!
